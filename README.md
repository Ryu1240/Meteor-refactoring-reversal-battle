# 逆リファクタリングバトル

## コンテスト目的・テーマ

### 目的

“正しく動く” FizzBuzz を題材に、どれだけ悪い設計・読みにくい構造を意図的に作れるかを競うコンテスト。

なぜそれが「悪い」のかを学ぶことで、日常開発の改善につなげる。

フォーマッタを通しても残る 本質的な可読性の低さ を探求する。

## ルール

### 対象言語

- Ruby
- TypeScript

### 実行・評価環境

評価はrubocop/biomeを使用してフォーマッタ適用後のコードを基準とする

### 評価基準（100点満点）

- 動作の正しさ
  - テストがすべて通ることが前提。
- 逆リファクタリング度
  - 構造・抽象化・フローの“ひどさ”。
- 学び・説明性
  - どのアンチパターンを意図したか、説明の明瞭さ。
- 芸術点
  - アイデアの面白さ、驚き、ネタ性。

### コンテストの流れ

1. ルール説明 & ベースコード配布
   -  参加者は言語を選び、配布されたベースコードとテストコードを受け取る。
2. 実装フェーズ
   - 制限時間内に逆リファクタリング作品を作成。
3. 提出
   - 指定方法で提出（zip / GitHub / 共有リンクなど）
4. 発表・講評
   - 提出者による意図説明
   - みんなでコメント
5. 結果発表
   - 最優秀賞・芸術点賞などを選出

### 禁止事項

■ ミニファイ行為（本質的でない読みづらさ）
- 自動ミニファイツールの使用
- すべて1行化するだけの実装

■ 競技の前提を壊す行為
- テストコードの改変
- 公開インターフェース（関数名・引数・戻り値）の変更
- フォーマッタ設定の変更

■ テストの実行速度を著しく低下させる行為
- sleepなどの遅延を含む実装
- 無限ループ
- 大量のデータ処理


## 逆リファクタリングの例

- 過剰な抽象化（無駄なクラス階層、無意味なインターフェイス）
- 状態の謎の持ち回り（グローバル依存、余計なオブジェクト生成）
- 追跡困難な制御フロー（例外利用、ネスト過多、無駄な非同期化）
- メタプログラミング乱用（eval、with、Proxy、method_missing など）
